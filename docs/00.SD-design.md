## 二、部署架構

系統部署於 **Vercel**，使用 Neon Serverless PostgreSQL 作為資料庫。

```
員工瀏覽器 ──→ Vercel（Next.js） ──→ Neon Postgres
                    │
                    └──→ NVD CVE API（弱點掃描時）
```

- 員工直接透過瀏覽器存取系統，上傳軟體清單
- Admin 在系統內觸發批次弱點掃描，系統直接查詢 NVD
- 所有資料儲存於 Neon Postgres，無需手動搬移

## 三、資料流向

```
【員工】
  執行 esrm-agent.ps1 掃描本機軟體
    → 上傳 JSON 至系統
    → 寫入 software_info + software_entries 表

【Admin】
  在「弱點掃描」頁面觸發批次掃描
    → 系統查詢 NVD CVE API
    → 結果寫入 vuln_cache + cve_entries 表

【所有人】
  登入後依角色查看軟體清單（自動合併弱點快取）
    → 下載 CSV / Excel 報表
```

## 五、資料庫設計（Drizzle Schema）

### 連線策略

| 環境 | 資料庫 | 連線 |
|------|--------|------|
| Vercel（生產） | Neon Serverless Postgres | `@neondatabase/serverless` |
| 本地開發 | Neon（遠端，同一個雲端 DB） | `@neondatabase/serverless` |

Drizzle 透過 `DATABASE_URL` 連線，Schema 定義統一為 PostgreSQL dialect。  
無論本地或 Vercel 部署，一律使用 Neon 雲端資料庫，不使用本地 PostgreSQL。

### 主要資料表

```
departments        → id, name, directorUsername, managerUsernames[]
groups             → id, name, deptId, leaderUsername, managerUsername, memberUsernames[]
users              → username, displayName, passwordHash, role, deptId, groupId, createdAt
software_info      → username, hostname, uploadedAt（每人一筆，upsert）
software_entries   → id, username, name, version, publisher, installDate, installPath
vuln_cache         → softwareKey（PK = "{name}|{version}"）, severity, cveIds, checkedAt
cve_entries        → id, softwareKey, cveId, cvss, description, publishedAt
scan_progress      → id=1（單筆）, status, total, completed, failed, currentSoftware, startedAt
```

### 弱點快取 Key

格式：`"{softwareName}|{version}"`，例如 `"Google Chrome|124.0.6367.82"`

**必須透過 `buildKey()` 產生，禁止自行拼接字串：**
```typescript
// src/lib/software-key.ts
export function buildKey(name: string, version: string): string
export function buildKey(entry: { name: string; version: string }): string
```

## 六、認證設計（NextAuth.js v5）

### 認證方式

使用 NextAuth Credentials Provider + bcrypt 比對資料庫密碼雜湊。使用者帳號由 Admin 在後台建立。

| 功能 | 設計 |
|------|------|
| 帳號驗證 | NextAuth Credentials Provider + bcrypt |
| 角色屬性 | `users.role` 欄位 |
| 部門屬性 | `users.deptId` 欄位 |
| 組屬性 | `users.groupId` 欄位 |
| 帳號建立 | Admin 在「使用者管理」頁面手動建立 |

### Session 結構

```typescript
interface EsrmSession {
  user: {
    username: string
    displayName: string
    role: 'ROLE_ADMIN' | 'ROLE_DIRECTOR' | 'ROLE_MANAGER' | 'ROLE_LEADER' | 'ROLE_MEMBER'
    deptId?: string
    groupId?: string
  }
}
```

### 路由保護

`middleware.ts` 攔截所有非登入頁的請求：
- 未登入 → 導向 `/login`
- 已登入但角色不足 → 導向 `/forbidden`（Next.js 15 內建 `forbidden.tsx` 慣例）
- Server Action 內再次驗證角色（Defense in Depth）

### 角色階層

```
ADMIN > DIRECTOR > MANAGER > LEADER > MEMBER
```

角色繼承邏輯透過輔助函式實作：

```typescript
// src/lib/auth.ts
export function hasRole(session: EsrmSession, minRole: EsrmRole): boolean
```

## 九、esrm-agent.ps1 輸出格式規格

`esrm-agent.ps1` 掃描完成後產生以下 JSON 格式，供 `parseAgentJson()` 解析：

```json
{
  "generatedAt": "2026-05-13T10:30:00.000Z",
  "hostname": "DESKTOP-ABCD123",
  "username": "vik",
  "softwares": [
    {
      "name": "Google Chrome",
      "version": "124.0.6367.82",
      "publisher": "Google LLC",
      "installDate": "2024-03-15",
      "installPath": "C:\\Program Files\\Google\\Chrome\\Application"
    }
  ]
}
```

**驗證規則**（`parseAgentJson()` 必須檢查）：
- 頂層必須有 `generatedAt`、`hostname`、`username`、`softwares` 四個欄位
- `softwares` 必須是陣列（可為空陣列）
- 每筆 software 必須有 `name` 與 `version`（非空字串）
- 格式不符時拋出錯誤，由 Route Handler 回傳 400

## 十、弱點掃描（Vercel Serverless 限制因應）

Vercel Serverless Function 預設 timeout 為 10 秒（Pro 可達 300 秒）。批次掃描需分段處理：

### 掃描架構

```
POST /api/vuln/scan（start）
  → 寫入 scan_progress（status: RUNNING, total: N）
  → 回傳立即回應

GET /api/vuln/progress（前端每 2 秒輪詢）
  → 回傳 scan_progress 目前狀態

POST /api/vuln/scan（batch）
  → 前端在收到進度後持續呼叫，直到 completed + failed = total
  → 每批次查詢 NVD API → 寫入 vuln_cache
  → 更新 scan_progress.completed / failed
```

- 有 API Key：每筆間隔 700ms
- 無 API Key：每筆間隔 6000ms（NVD 速率限制）
- 單筆失敗記錄 failed 計數，不中斷整體掃描

## 十三、除錯速查

| 症狀 | 排查方向 |
|------|----------|
| NextAuth 登入失敗 | 確認 `AUTH_SECRET` 已設定；Credentials callback 錯誤訊息看 server log |
| Drizzle 找不到資料表 | 執行 `pnpm db:push` 或 `pnpm db:migrate` |
| Server Action 無效 | 確認函式最頂行有 `'use server'` |
| Middleware 無法讀取 session | `middleware.ts` 使用 `auth()` wrapper，確認 `AUTH_SECRET` 一致 |
| Vercel 函式 timeout | 掃描批次改小；或升級 Vercel Pro 延長 timeout |
| Neon connection 錯誤 | 確認 `DATABASE_URL` 包含 `channel_binding=require&sslmode=require` |
| 弱點按鈕不顯示 | 確認使用者角色為 ROLE_ADMIN |
| drizzle-kit 無法讀取 DATABASE_URL | `drizzle.config.ts` 需手動載入 `.env.local`（見第十四節） |

## 十四、Neon 資料庫建立與 Drizzle 說明

### 14.1 Drizzle ORM 是什麼

Drizzle 是一個 TypeScript ORM（物件關聯映射）工具，讓你用 TypeScript 程式碼操作資料庫，不需要直接撰寫 SQL。

| 套件 | 角色 |
|------|------|
| `drizzle-orm` | 執行期：連線、查詢、插入、更新 |
| `drizzle-kit` | 開發工具：`db:push`、`db:studio`、產生 migration |

**與 Next.js 的分工**：Next.js 會自動載入 `.env.local`，但 `drizzle-kit` 是獨立 CLI 工具，**不會自動讀取 `.env.local`**。因此 `drizzle.config.ts` 需要手動用 `dotenv` 載入：

```typescript
import { config } from 'dotenv'
config({ path: '.env.local' })  // 必須加在最頂部
```

同理，`seed.ts` 透過 `tsx` 直接執行時也需要手動載入，已在檔案頂部加入相同處理。

### 14.2 Neon DB 建立方式（透過 Vercel 整合）

**建立步驟（一次性設定，無需另外到 neon.tech 註冊）**：

1. Vercel Dashboard → 你的專案 → **Storage** 頁籤
2. 點擊 **Connect Database** → 選擇 **Neon**
3. 建立資料庫（填入名稱，選擇 Region）
4. Vercel 自動將以下環境變數注入到專案：
   - `DATABASE_URL`（pooled 連線，供應用程式使用）
   - `POSTGRES_URL_NON_POOLING`（non-pooled，供 migration 使用）

**取得本地開發的連線字串**：

Vercel Dashboard → Storage → 選擇 Neon DB → **Connection Details** → 複製 `DATABASE_URL`

連線字串格式（Neon 實際格式，包含 `channel_binding`）：
```
postgresql://user:pass@ep-xxx-pooler.region.aws.neon.tech/neondb?channel_binding=require&sslmode=require
```

### 14.3 資料庫分支建議（開發 / 正式隔離）

Neon 支援類似 Git 的 Branch 功能，建議規劃如下：

```
Neon Project
├── main branch    → Vercel Production 使用
└── dev branch     → 本機開發使用（測試資料不污染正式環境）
```

建立方式：Neon Dashboard → 你的 Project → **Branches** → **Create Branch**

本機 `.env.local` 填入 `dev` branch 的連線字串；Vercel Production 自動使用 `main` branch。

### 14.4 drizzle-kit 常用指令說明

| 指令 | 說明 | 使用時機 |
|------|------|---------|
| `pnpm db:push` | 直接將 `schema.ts` 同步到資料庫 | 開發階段快速迭代 |
| `pnpm db:generate` | 根據 schema 變更產生 migration SQL | 正式環境變更前 |
| `pnpm db:migrate` | 執行 migration 檔案 | 正式環境部署時 |
| `pnpm db:studio` | 開啟網頁版資料庫瀏覽器 | 查看 / 手動編輯資料 |
| `pnpm db:seed` | 插入預設測試帳號與組織資料 | 初始化或重置測試環境 |
