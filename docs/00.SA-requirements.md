# ESRM 系統需求分析文件
# Enterprise Software Risk Monitor — System Analysis

**文件版本**：2.0  
**撰寫日期**：2026-05-13  
**文件性質**：系統分析（SA）—— 功能需求，不涉及技術實作

---

## 一、系統背景與目的

### 1.1 問題描述

企業員工電腦裝有各式軟體，部分版本存在已知資安漏洞（CVE），若未及時更新，可能成為資安攻擊的入口。資安人員難以掌握全公司的軟體風險現況。

### 1.2 系統目標

1. **掌握全公司軟體資產**：收集每位員工電腦安裝的軟體名稱與版本。
2. **識別已知弱點**：對應外部 CVE 弱點資料庫，找出有已知漏洞的軟體版本。
3. **提供弱點警示**：讓各層主管及員工清楚看到自己或所屬團隊的軟體風險狀態。
4. **支援報表輸出**：提供 CSV、Excel 報表供資安人員進行後續風險管理。

### 1.3 部署架構

系統部署於 **Vercel 雲端**，員工透過瀏覽器存取，無需本地安裝。系統直接查詢 NVD CVE 資料庫，不需要在不同環境間手動傳遞資料。

## 二、使用環境說明

### 2.1 系統架構

| 特性 | 說明 |
|------|------|
| 部署環境 | Vercel 雲端 |
| 資料庫 | Neon Serverless PostgreSQL |
| 網路要求 | 員工需能存取系統 URL |
| 主要使用者 | 全體員工、各層主管、系統管理員 |

### 2.2 資料流程

```
【員工】執行 esrm-agent.ps1 → 掃描本機軟體 → 上傳 JSON → 系統儲存
【Admin】在系統觸發批次弱點掃描 → 系統直接查詢 NVD CVE 資料庫 → 結果寫入 DB
【所有人】登入系統 → 依角色查看軟體清單與弱點狀態 → 下載報表
```

## 三、使用者角色

系統共有五個角色，採階層式繼承：**高階角色自動擁有低階角色的所有權限**。

| 角色 | 說明 | 可查看的軟體範圍 |
|------|------|----------------|
| **系統管理員（Admin）** | 全系統最高權限，負責組織管理、弱點掃描 | 全公司所有人 |
| **協理（Director）** | 部門最高主管 | 本部門所有人 |
| **經理（Manager）** | 負責部門下一個或多個組 | 本部門所有人 |
| **組長（Leader）** | 管理一個組的成員 | 本組所有人 |
| **一般成員（Member）** | 一般員工 | 僅限本人 |

---

## 四、組織結構

系統管理一套三層式組織架構：

```
公司
└── 部門（Department）
    ├── 協理（1 人）
    ├── 經理（1 人以上）
    └── 組（Group）
        ├── 組長（1 人）
        └── 組員（0 人以上）
```

**重要業務規則**：
- 每位員工只能屬於**一個組**。
- 組長帳號不計入組員清單。
- 刪除部門前，必須先刪除該部門下的所有組。
- 刪除組前，必須先移除組內所有成員。

## 五、功能需求

### 5.1 組織管理（Admin 專用）

管理員可對公司組織架構進行維護：

**部門管理**
- 新增部門（填入部門名稱、指定協理帳號）
- 修改部門名稱、變更協理、調整負責經理
- 刪除部門（前提：部門下無組）

**組管理**
- 在指定部門下新增組（填入組名稱、負責經理、組長）
- 修改組名稱、組長、負責經理
- 刪除組（前提：組內無成員）

**成員管理**
- 將帳號加入指定組的成員清單
- 從組中移除成員
- 同一帳號不可同時存在於兩個組的成員清單中

---

### 5.2 使用者管理（Admin 專用）

Admin 在系統後台建立與管理使用者帳號：

- 新增使用者（帳號、顯示名稱、密碼、角色、部門、組）
- 修改使用者角色與組織歸屬
- 重設密碼
- 停用 / 刪除帳號

---

### 5.3 軟體資訊收集

#### 5.3.1 掃描工具下載

系統提供一支掃描腳本（`esrm-agent.ps1`），員工可在「我的軟體清單」頁面下載。

#### 5.3.2 員工上傳軟體清單

1. 員工在自己的 Windows 電腦執行掃描腳本。
2. 腳本掃描已安裝的軟體（名稱、版本、發行商、安裝日期、安裝路徑），產生一份 JSON 檔案。
3. 員工登入系統，上傳該 JSON 檔案。
4. 系統儲存後顯示個人軟體清單。

**業務規則**：
- 每位員工的軟體資訊**只保留最新一份**；重複上傳時，舊資料直接被覆蓋。
- 上傳時僅接受正確格式的掃描腳本輸出，格式錯誤需提示使用者。

#### 5.3.3 軟體清單查看

依角色顯示對應範圍的軟體清單：

| 頁面 | 可查看角色 | 說明 |
|------|-----------|------|
| 我的軟體清單 | 所有人 | 本人所有已安裝軟體 + 弱點狀態 |
| 組軟體清單 | 組長以上 | 本組所有成員的軟體（含弱點） |
| 部門軟體清單 | 經理 / 協理以上 | 本部門所有成員的軟體（含弱點） |
| 全公司軟體清單 | 系統管理員 | 全公司所有人的軟體（含弱點） |

清單顯示項目：
- 軟體名稱、目前安裝版本、最新可用版本
- 弱點狀態（CRITICAL / HIGH / MEDIUM / LOW / 無弱點 / 尚未查詢）
- 點擊軟體可展開查看詳細 CVE 清單（CVE 編號、CVSS 分數、說明、發布日期）

---

### 5.4 弱點快取機制

弱點資訊**與使用者無關**，以「軟體名稱 + 版本」為識別鍵：
- 同一套軟體（相同名稱與版本），全公司共用一份弱點資訊。
- 未掃描前，該軟體的弱點狀態顯示為「尚未查詢」。
- 每次掃描時，會**覆蓋更新**已有的弱點資訊。

---

### 5.5 弱點查詢（Admin 專用）

#### 5.5.1 批次弱點掃描

- 執行角色：**系統管理員**
- 操作：在「弱點掃描」頁面啟動批次掃描
- 系統行為：
  1. 取得所有已上傳的軟體清單，去除重複（相同名稱 + 版本視為同一筆）
  2. 逐一查詢外部 CVE 資料庫（NVD）
  3. 即時更新查詢進度（進度條、已完成數量、目前查詢中的軟體名稱）
  4. 查詢失敗的軟體記錄失敗次數，繼續查詢其他軟體（不因單筆失敗而中斷整體掃描）

#### 5.5.2 查詢速率控制

為避免超出外部資料庫的查詢次數限制，系統自動控制查詢間隔（有 API 金鑰時較快，無金鑰時較慢）。

#### 5.5.3 快取避免重複查詢

已查詢過且在有效期限內的軟體，不重複呼叫外部資料庫。

#### 5.5.4 取消掃描

掃描進行中，管理員可隨時中止。

### 5.6 儀表板

登入後首頁依角色顯示不同的統計摘要：

| 角色 | 顯示內容 |
|------|---------|
| 系統管理員 | 全公司人數、已上傳軟體人數、有弱點軟體套數、弱點資料最後掃描時間 |
| 協理 / 經理 | 本部門人數、已上傳人數、有弱點軟體套數 |
| 組長 | 本組人數、已上傳人數、有弱點軟體套數 |
| 一般成員 | 個人軟體套數、有弱點軟體套數、資料最後上傳時間 |

---

### 5.7 報表功能

#### 5.7.1 報表種類

| 報表 | 內容 | 可使用角色 |
|------|------|-----------|
| 個人軟體報表 | 個人所有軟體 + 弱點狀態 | 所有人 |
| 組別軟體報表 | 全組成員軟體 + 弱點 | 組長以上 |
| 部門軟體報表 | 全部門成員軟體 + 弱點 | 經理 / 協理以上 |
| 全公司軟體報表 | 全公司所有人軟體 + 弱點 | 系統管理員 |

#### 5.7.2 報表欄位

| 欄位 | 說明 |
|------|------|
| 使用者帳號 | 員工帳號 |
| 部門 / 組別 | 所屬部門與組名稱 |
| 主機名稱 | 員工電腦名稱 |
| 最後上傳時間 | 軟體清單最後更新時間 |
| 軟體名稱 | 已安裝軟體名稱 |
| 安裝版本 | 目前安裝的版本號 |
| 最新版本 | 官方最新版本（若有資料） |
| 發行商 | 軟體發行商名稱 |
| 安裝日期 | 安裝於電腦的日期 |
| 是否有弱點 | 有 / 無 / 尚未查詢 |
| 最高弱點等級 | CRITICAL / HIGH / MEDIUM / LOW |
| CVE 清單 | 相關 CVE 編號（逗號分隔） |
| CVSS 最高分 | 最嚴重弱點的 CVSS 評分 |
| 弱點最後查詢時間 | 弱點資料最後更新時間 |

#### 5.7.3 報表格式

- **CSV**：純文字格式，適合 Excel 開啟與進一步分析
- **Excel**：含格式化（有弱點列依嚴重程度標示顏色）、包含「詳細清單」與「統計摘要」兩個工作表

#### 5.7.4 權限規則

- 一般成員只能下載個人報表。
- 組長只能下載本組及個人報表，不得下載其他組。
- 經理 / 協理只能下載本部門及以下範圍的報表。
- 系統管理員可下載所有範圍報表。

## 六、認證與安全

### 6.1 登入方式

- 使用帳號 + 密碼登入，密碼以 bcrypt 雜湊存於資料庫。
- 登入失敗顯示錯誤訊息，但不透露帳號是否存在。
- Session 採 JWT 機制，存於 HttpOnly Cookie。

### 6.2 使用者資訊來源

- 使用者的帳號、角色、所屬部門/組，由 Admin 在系統後台建立。
- 開發/測試環境可使用 `pnpm db:seed` 插入預設測試帳號。

### 6.3 存取控制

- 每個頁面與操作均有對應的角色限制，未授權的存取返回「無存取權限」頁面。
- 登出後 Session 立即清除，不可再存取任何後台頁面。

## 七、業務規則彙整

| 規則 | 說明 |
|------|------|
| 軟體資訊唯一性 | 每位員工只保留最新一份軟體清單，上傳即覆蓋 |
| 弱點資訊共用性 | 相同軟體名稱 + 版本，全公司共用同一份弱點資訊 |
| 組員唯一歸屬 | 每位帳號只能屬於一個組 |
| 角色繼承 | 高階角色自動擁有低階角色的所有功能 |
| 查詢快取 | 已查詢且在有效期限內的弱點資料不重複查詢 |

## 八、非功能需求（概述）

| 需求類別 | 說明 |
|----------|------|
| 雲端部署 | 系統部署於 Vercel，無需本地安裝，員工有瀏覽器即可使用 |
| 資料持久性 | 使用 Neon Serverless PostgreSQL，資料自動備份 |
| 靜態資源 | 所有 UI 樣式、腳本均透過 npm 套件管理，不依賴外部 CDN |
| 資料安全 | 員工軟體清單及弱點資訊屬於企業敏感資料，應限制存取 |
| 可擴展性 | 系統設計支援數百名員工同時使用 |
